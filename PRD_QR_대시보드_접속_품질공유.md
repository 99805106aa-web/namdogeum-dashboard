# PRD — QR 코드를 통한 대시보드 접속 및 품질 정보 공유
**남도금형 스마트 품질 모니터링 대시보드**  
작성일: 2026-04-24 | 버전: 0.1 (초안)

---

## 1. 배경 및 목적

### 1.1 배경
현재 품질 모니터링 대시보드(`dashboard.html`)는 브라우저에서 URL을 직접 입력하거나 북마크로 접속하는 방식에 의존한다.  
현장·협력사·경영진 간 **품질 현황을 빠르게 열람·공유**하려면, 링크 복사·전달의 마찰을 줄이는 접근이 필요하다.  
QR 코드는 모바일에서 즉시 브라우저를 열 수 있어, **인쇄물·설비·납품 포장** 등에 부착해 “스캔 → 동일 품질 화면”으로 이어지는 경로를 만드는 데 적합하다.

### 1.2 목적
1. **QR 스캔 한 번**으로 품질 대시보드(또는 합의된 뷰)에 접속할 수 있게 한다.  
2. 접속자(내부/외부)에게 **정해진 범위의 품질 정보**를 안전하게 공유할 수 있는 정책을 정의한다.  
3. 기존 단일 HTML 대시보드 구조를 과도하게 바꾸지 않는 범위에서 **단계적 구현**이 가능하도록 한다.

### 1.3 기대 효과
| 항목 | 현재 | 개선 후 |
|------|------|---------|
| 접속 수단 | URL 수동 전달 | QR 스캔으로 즉시 이동 |
| 현장/외부 공유 | 링크 복사·메신저 의존 | 인쇄 QR·스티커·간판으로 고정 |
| 인지도 | “어디서 보는지” 불명확 | 오프라인 터치포인트와 연결 |
| 보안/범위 | (미정) 전체 URL 노출 | 단계에 따라 토큰/역할로 제한 가능 |

## 2. 사용자·시나리오

### 2.1 주요 페르소나
| 페르소나 | 니즈 | 제약 |
|----------|------|------|
| 현장/생산 | 설비 옆에서 최신 품질 지표 확인 | 모바일, 빠른 로딩 |
| 품질 담당 | 협력사·내부에 “이 주소로 보라”고 안내 | 공개 범위 조절 |
| 경영/내부 | 회의실·보고서에 QR로 접속 | 내부망/외부망 이슈 |
| (선택) 협력사/고객 | 계약/납품 단위 품질 요약 | 민감 데이터 노출 방지 |

### 2.2 핵심 시나리오
1. **S1 — 내부 전용**  
   사내 Wi-Fi 또는 VPN 전제. QR은 **고정 URL**(예: 내부 HTTPS 게이트웨이 뒤의 `/dashboard.html`)로 연결. 인증은 기존 SSO/베이직 인증이 있으면 그대로.

2. **S2 — 인쇄물·라벨**  
   주간보고 PDF·납품 라벨에 QR을 삽입. 스캔 시 **동일 품질 요약 뷰** 또는 `?view=summary`와 같은 쿼리로 **탭 제한**된 화면.

3. **S3 — 제한된 외부 공개**  
   URL에 **만료/회전 토큰** 또는 **읽기 전용 게스트 토큰**을 붙인 QR. 유출 시 재발급·무효화.

## 3. “공유”의 정의 (범위 확정)

아래 중 제품이 지원할 **공유 수준**을 1~2가지로 우선 결정한다.

| 레벨 | 설명 | 구현 복잡도 |
|------|------|-------------|
| L0 | URL만 QR로 인코딩. 접근 제어 없음(비권장, 내부망 전용) | 낮음 |
| L1 | 공개 **읽기 전용** 스냅샷(특정 기간·KPI만). 정적 HTML 또는 서버가 만든 단순 페이지 | 중간 |
| L2 | 토큰+만료(예: 7일)가 붙은 링크. 토큰 없으면 403 | 중~높음 |
| L3 | 로그인·역할(내부/협력사)별로 탭/데이터 가림 | 높음 |

**본 PRD 권장:** 1단계 **L0 또는 L1 + 내부 호스팅**으로 PoC, 보안 요구가 있으면 **L2**로 승격.

## 4. 기능 요구사항

### 4.1 Must Have (1차)
| ID | 요구사항 | 비고 |
|----|----------|------|
| R-QR-01 | 대시보드 **접속 URL**을 QR 이미지로 생성·다운로드(또는 인쇄)할 수 있다. | 서버/클라이언트 라이브러리 또는 외부 API |
| R-QR-02 | QR에 들어갈 **기준 URL**이 배포 환경(로컬/스테이징/운영)에 맞게 설정 가능하다. | 환경 변수 또는 설정 JSON |
| R-QR-03 | 모바일 브라우저에서 스캔 시 **뷰포트·탭**이 사용 가능(기존 `dashboard.html`의 반응형·햄버거 메뉴와 모순 없음) | 기존 UI 회귀 테스트 |
| R-QR-04 | (선택) `?kiosk=1` 등으로 **최소 UI**(사이드바 기본 닫힘) 모드 | 현장 디스플레이용 |

### 4.2 Should Have (2차)
| ID | 요구사항 | 비고 |
|----|----------|------|
| R-QR-05 | 주간보고/내보내기(PDF) 시 **같은 기간·같은 공개 수준**의 QR이 함께 찍힌다. | `html2canvas`/jsPDF 흐름과 연동 |
| R-QR-06 | “공개용 링크 발급” UI에서 **만료일·한 줄 설명**을 입력해 토큰 URL을 생성. | L2 이상 |
| R-QR-07 | 토큰 **폐기·재발급** 관리(관리자) | L2 이상, 백엔드 필요 |

### 4.3 Could Have
- NFC 태그와 동일 URL 병행  
- 스캔 **로그(선택)**: IP/시간만 저장해 접속 통계  
- 품목·라인별 **딥링크** QR (`#tab=...` 또는 쿼리)

### 4.4 Won’t (초기 제외)
- QR만으로 “데이터 편집” 권한 부여  
- 앱 전용 딥링크(스토어) 없이 PWA가 필수인 요구

## 5. 비기능 요구사항

| 구분 | 내용 |
|------|------|
| 성능 | QR 리다이렉트는 **1-hop**(QR → HTTPS URL)로 끝낼 것. 불필요한 중간 리다이렉트 최소화. |
| 보안 | HTTPS. 외부 공개 시 **최소 데이터**·**만료**·**재발급** 원칙. |
| 가용성 | URL이 가리키는 호스트가 죽으면 QR 전체가 무의미 → **SLA/모니터링**은 운영 정책에 포함. |
| 접근성 | QR 옆에 **짧은 URL 텍스트**를 병기(접근성·수동 입력 대비). |
| 국제화 | 오류 메시지·만료 안내 **한국어** 우선(기존 앱과 동일). |

## 6. 기술 구현 방안 (How)

현재 코드베이스가 **단일 `dashboard.html` + 클라이언트 JS**이므로, 아래를 조합해 선택한다.

### 6.1 QR 생성 측
- **A. 완전 클라이언트**  
  - 예: `qrcode`/`qr-code-styling` 등 **브라우저 JS**로 캔버스에 QR 생성, 이미지 다운로드.  
  - 장점: 백엔드 불필요. 단점: “비밀 토큰 URL”을 클라이언트에서만 만들면 유출 시 대응이 어려움(비밀은 서버가 발급하는 편이 안전).  
- **B. 백엔드가 서명·발급**  
  - `/api/qr-link`가 토큰 URL 반환, 프론트는 그 URL로 QR 이미지 생성.  
  - L2 이상·감사 로그가 필요할 때.

### 6.2 URL 설계
- **고정 뷰**  
  - `https://{host}/quality/dashboard.html`  
- **기간/탭 제한(쿼리)**  
  - `?from=2026-04-01&to=2026-04-30&view=executive`  
- **비밀·만료(토큰)**  
  - `?t={jwt 또는 opaque token}` — 검증은 **서버** 또는 **정적 키로 검증 가능한 JWS** (키 관리 주의).  
- **주의**  
  - `file://` QR은 기기마다 동작이 달라 **권장하지 않음**. 반드시 **https:// 호스트**에 올릴 것.

### 6.3 배포
- 정적 파일 호스팅(예: 사내 Nginx, S3+CloudFront, GitHub Pages 등) + TLS.  
- CORS/캐시: 대시보드가 외부 API를 부르면 CORS·API 키 정책을 QR 시나리오에 맞게 재검토.

### 6.4 데이터 민감도
- 엑셀/로컬 스토리지·업로드 데이터가 있다면, **“공개 QR”**이 **어떤 데이터를 서버/클라이언트에 남기는지**를 데이터 보호 검토에 포함.

## 7. 수용 기준 (Acceptance Criteria) — 1차 PoC

1. **AC1**  
   설정된 `BASE_URL`로 인코딩된 QR을 화면에서 생성해 PNG로 저장할 수 있다.  
2. **AC2**  
   iOS·Android 기본 카메라로 스캔 시 **같은 대시보드**가 열린다(동일 URL 전제).  
3. **AC3**  
   `BASE_URL`을 바꾸면 QR 내용이 그에 맞게 바뀐다(환경 분리).  
4. **AC4**  
   내부 정책에 따라, 외부에 나가는 QR이면 **L1/L2** 요구를 만족하는지 별도 체크리스트로 승인.

## 9. 리스크·오픈 이슈

| 이슈 | 대응 방향 |
|------|-----------|
| URL이 유출되면 누구나 접속(L0) | 내부망/VPN, 또는 L2로 전환 |
